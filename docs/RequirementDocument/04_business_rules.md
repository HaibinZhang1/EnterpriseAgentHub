# 5. 核心业务规则

> 阶段说明：本章为完整产品蓝图规则。P1 Desktop 需要遵守可见、可安装、安装/启用、版本覆盖、权限收缩的消费侧规则；发布、审核、权限变更审批等写入型治理规则进入 P2。

## 5.1 发布、可见、可安装分离
skill 的"能否被看见""能否查看详情""能否安装/使用"必须分离管理：

- 授权范围：决定谁能安装/使用
- 公开级别：决定谁能看到摘要或详情
- 安装与启用资格：最终由授权范围和公开级别共同决定

## 5.2 推荐的默认策略
默认采用以下策略：

- 授权范围：由发布者选择
- 公开级别：默认不公开，仅授权范围内可见
- 审核结果：仅对可查看详情的用户展示完整信息
- 全站曝光榜：仅作为系统排行榜能力，默认展示前 20 名

## 5.3 全站曝光榜
为了满足"想曝光但不暴露细节"的诉求，支持全站曝光榜能力，但其本质是**系统排行榜**，不是单独的权限级别。

规则如下：
- 默认仅展示前 20 名
- 榜单是否可见，取决于 skill 的公开级别
- 摘要公开 skill 在榜单中仅展示摘要信息
- 详情公开和全员可安装 skill 可展示完整详情入口
- 非公开 skill 不进入全站曝光榜

## 5.4 审核结果公开范围
对于可查看详情的用户，详情页展示：
- 审核通过时间
- 审核人
- 风险等级
- 审核意见摘要
- 系统初审结果

对于仅可见摘要的用户：
- 可显示"已审核发布"标识
- 可选显示风险等级标签（低 / 中 / 高）
- 不展示审核意见摘要与审核人

## 5.5 版本更新规则
- 版本号需遵循 **SemVer**（语义化版本）规范，格式为 `MAJOR.MINOR.PATCH`
- 版本号必须递增，**不支持预发布版本号**（如 1.0.0-beta.1）
- 版本更新流程与首次发布流程一致（统一状态机，通过字段区分"首次发布/更新发布"）
- 审核链路按实时组织链路重新计算
- 已发布旧版本在新版本审核期间仍保持可用
- 用户通过市场安装后，更新时系统校验本地文件 Hash 值
- 若发现本地已被修改，则提示"本次更新将直接覆盖本地内容"
- 系统采用**直接覆盖策略**，不提供"保留本地修改再启用"的能力

## 5.6 发布者操作规则
- 发布者可以撤回自己待审核中的 skill（审核中/已锁单状态除外，详见 [生命周期](05_lifecycle_states.md)）
- 发布者可以删除自己已发布的 skill（逻辑归档）
- 删除后整个 skill 历史版本统一归档

## 5.7 权限变更规则
已发布 skill 支持修改授权范围和公开级别，但需遵循以下规则：

### 5.7.1 审核要求（差异化处理）
- **权限扩大**（如部门内 → 全员可安装）：按扩大后的公开级别重新计算审核链路
- **权限收缩**（如全员 → 部门内）：简化审核，仅需同级管理员确认即可
- 权限变更走独立的权限变更状态流，不混入 skill 版本发布状态
- 权限变更审核期间，**原权限配置继续生效**
- 权限修改不变更 skill 代码版本号，但产生权限变更记录

### 5.7.2 对存量用户的影响
系统采用统一原则：

**存量继续使用，但禁止新增安装与更新。**

为避免歧义，具体解释如下：
- "存量用户"按**是否已经安装过该 skill**判定
- 已安装且已启用的用户：可继续使用当前版本，不可更新
- 已安装但未启用的用户：可继续启用当前已安装版本，不可更新
- 未安装用户：权限收缩后不可新增安装，也不可新增启用
- 已卸载的用户：视为非存量用户，不能再次安装
- 存量用户不得通过更新或重新安装的方式绕过权限收缩

### 5.7.3 对启用位置的影响
- 已启用位置维持原状，不自动停用
- 已安装但未启用的本地副本，可继续在当前设备内启用
- 不支持借由复制本地文件到其他工具目录、项目目录来新增实际可用范围
- 权限收缩后新增启用范围、重新安装、版本更新均禁止

> 说明：
> 上述规则用于消解"存量继续使用"与"禁止新增"之间的冲突：
> 这里允许的是**已安装副本的延续使用**，不允许的是**新增获得能力**。
