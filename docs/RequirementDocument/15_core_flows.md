# 10. 核心流程说明

> 说明：本页同时记录发布治理流程与 Desktop 本机安装/启用流程；当前仓库已经包含真实作者、审核、管理与本机闭环实现。

## 10.1 Skill 发布流程
1. 用户进入"发布 Skill"
2. 上传文件夹或 zip
3. 系统解析 skill 元信息与包结构
4. 用户补充 / 修改字段
5. 客户端完成可前置的发布预检查，服务端执行最终校验
6. skill 上传到服务器（通过服务端中转）
7. 系统自动生成 manifest.json
8. 进入"系统初审中"
9. 初审通过则进入"待管理员审核"，写入 `system_precheck_passed` 历史，并通知作者与最近一级审核员
10. 初审发现异常则进入"待人工复核"，写入 `system_precheck_needs_manual_review` 历史，并通知作者与最近一级审核员
11. 对应管理员审核（审核链路按实时组织链路计算），审核员领取后进入"审核中"，并记录领取来源
12. 审核同意：
    - 状态变为"已发布"
    - skill 进入市场
    - 写入 `approve` / `auto_approve` 历史，并通知作者
13. 审核拒绝：
    - 状态变为"审核拒绝"
    - 用户可查看拒绝原因，可重新提交
    - 写入 `reject` 历史，并通知作者
14. 退回修改：
    - 状态变为"退回修改"
    - 用户可修改后重新提交
    - 写入 `return_for_changes` 历史，并通知作者

发布审核链路的用户提醒使用 `notifications`：作者进度通知指向发布单，审核任务通知指向审核单。通知不因审核员领取而撤回，点击后以服务端实时详情状态为准。

## 10.2 Skill 版本更新流程
与首次发布流程一致（统一状态机，通过 `publish_type: update` 字段标记），但：
- 审核链路按实时组织链路重新计算
- 已发布旧版本在新版本审核期间保持可用
- 版本号必须递增（SemVer）

## 10.3 权限变更流程
- 权限范围 / 公开级别修改走独立的权限变更状态流
- 权限扩大：按扩大后的公开级别重新计算审核链路
- 权限收缩：简化审核，仅需同级管理员确认
- 权限变更不修改代码版本号，也不更新分类、标签、兼容工具或包文件元数据，但产生权限变更记录
- 审核期间原权限配置继续生效

## 10.4 撤回流程
- 发布者可对以下状态的 skill 执行撤回：上传成功、系统初审中（终止扫描）、待人工复核、待管理员审核
- **审核中（已锁单）不可撤回**
- 撤回后状态变为"已撤回"，写入 `withdrawn` 历史，并通知作者

## 10.5 下架流程
- 已发布 skill 可被发布者或管理员下架
- 下架后市场不再展示 / 安装
- 已安装用户不受影响
- 跨部门紧急下架仅一级管理员可操作

## 10.6 上架流程
- 已下架 skill 可被发布者或管理员重新上架
- 上架后重新进入市场并恢复可安装资格
- 已归档 skill 不可重新上架
- 上架动作需保留状态变更记录

## 10.7 删除（归档）流程
- 发布者或管理员可执行删除（归档）
- 删除后整个 skill 历史版本统一归档
- 市场不再展示
- 不再作为活跃 skill 使用
- 已归档 skill 不可直接恢复；如需继续使用同一 slug，必须重新走首次发布审核

## 10.8 审核 / 提交详情文件预览流程
1. 作者或审核员进入详情页
2. 服务端返回当前提交包文件列表
3. 页面默认选中 `SKILL.md`，没有则选中第一个可预览文件
4. 若文件为 `.md` / `.markdown` / `.txt`，服务端返回文本内容并在页面中只读展示
5. 若文件不支持预览，则仅提供下载入口
6. 若文件内容超过 256 KB，服务端截断并返回截断标记

## 10.9 安装与启用流程
1. 用户在市场中安装 skill
2. 客户端申请安装用途下载凭证，服务端按已登录用户和 skill 记录一次去重安装下载
3. skill 通过**服务端中转**下载到本地 Central Store
4. 状态为"已安装未启用"
5. 用户选择工具 / 项目启用
6. 系统通过 Tool Adapter 做目标工具格式转换，并以 symlink 优先、失败自动降级 copy 的方式分发到对应工具/项目目录
7. skill 在对应工具 / 项目中生效
8. 市场与"已安装"页实时显示"已安装 / 已启用"
9. 如工具级与项目级冲突，则项目级优先
10. 如工具路径与项目路径冲突，则项目路径优先

## 10.10 卸载流程
1. 用户点击卸载
2. 系统提示当前被哪些工具 / 项目引用
3. 用户确认后直接卸载（从 Central Store 及所有目标目录移除）
4. 卸载后当前 skill 在本机不可继续使用
5. 已卸载用户后续若权限已收缩，则不能重新安装

## 10.10A 未托管本地 Skill 纳入管理流程
1. 用户扫描本地工具 / 项目 skills 目录
2. 系统仅将根目录包含 `SKILL.md` 的未托管目录展示为可纳入管理的 Skill
3. 用户点击“纳入管理”
4. 系统重新校验来源目录位于已配置目标根路径内，并解析 `SKILL.md`
5. 系统将来源目录复制到本机 Central Store，生成 `sourceType=local_import` 的本地安装记录
6. 系统默认认领原来源路径为已启用目标；同 skillID 且 checksum 一致的其他来源一并认领
7. 如 Central Store 已有同名 Skill 或扫描到同名不同内容来源，弹窗提示用户重命名导入、保留现有或确认替换
8. 纳入管理不上传服务器，不创建发布记录，不同步安装 / 更新事件
9. 本地导入项后续卸载时，确认后删除 Central Store 副本和已认领目标路径

## 10.11 客户端在线升级流程
1. 用户登录成功，客户端完成 `/desktop/bootstrap`
2. 客户端读取本机版本、构建号、平台、架构和设备 ID
3. 客户端调用服务端更新检查接口
4. 服务端按版本号、渠道、灰度比例、最低支持版本和发布状态返回检查结果
5. 无更新时，设置中展示“已是最新版本”
6. 有可选更新时，顶栏通知生成软件更新提示，设置模态展示更新卡片
7. 有强制更新或当前版本不受支持时，设置模态置顶提示，并限制发布、安装、审核、管理等远端写入型操作
8. 用户点击“立即更新”后，客户端申请短期下载凭证
9. 客户端下载完整安装包到受控临时目录
10. 下载完成后校验 SHA-256 和签名
11. 校验通过后，用户确认启动安装程序
12. 新版本启动并重新登录后，客户端上报已安装版本，更新提示消失

## 10.12 管理员客户端更新包推送流程
1. 管理员从可信 CI 或构建机取得已签名 Windows x64 安装包、SHA-256 文件和更新说明
2. 管理员将交付物上传到 Linux 服务器发布目录
3. 管理员在 Docker 环境中执行客户端更新包发布命令
4. API 容器校验版本号、Hash、签名状态和文件大小
5. API 将安装包写入 MinIO `client-updates` bucket
6. API 创建发布记录，并按渠道、灰度比例和是否强制更新生效
7. 管理员使用旧版本客户端或检查接口验证新版本可见
8. 灰度验证通过后，管理员扩大灰度比例至 100
9. 发现问题时，管理员将版本暂停或撤回，客户端下载凭证停止签发
