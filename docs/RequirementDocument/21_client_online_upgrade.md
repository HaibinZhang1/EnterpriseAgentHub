# 21. 客户端在线升级需求

## 1. 目标与范围

客户端在线升级用于让已登录用户及时获得桌面客户端新版本，并让系统管理员可以在 Linux Docker 部署的服务端推送 Windows 客户端更新包。

当前版本目标：

- 用户登录成功后自动检测桌面客户端是否存在新版本。
- 存在新版本时，在顶栏通知中生成软件更新通知，并在设置模态中展示更新提示。
- 用户可查看版本号、更新说明、包大小、发布时间和是否强制更新。
- 用户确认后下载完整安装包，客户端完成 Hash/签名校验后引导用户安装。
- 系统管理员可在 Linux 服务器的 Docker 环境中上传、发布、暂停、回滚客户端更新包。

当前版本边界：

- 首期只覆盖正式 Windows x64 客户端安装包；macOS/Linux 客户端更新包作为后续扩展。
- 首期采用完整安装包更新，不做差分补丁。
- 不做静默强制安装；即使是强制更新，也需要用户确认下载并按系统安装流程完成安装。
- 客户端在线升级只管理 Desktop/Tauri 客户端本身，不替代 Skill 包更新、服务端镜像升级或数据库迁移。
- 更新包必须由企业可信构建链路产出并完成 Windows 代码签名；服务端不在生产服务器上构建客户端安装包。

## 2. 角色与权限

| 角色 | 能力 |
| --- | --- |
| 普通用户 | 登录后接收更新提示、查看更新说明、下载并安装可用客户端版本、手动检查更新。 |
| 系统管理员 | 上传客户端更新包、填写版本元数据、发布到指定渠道、设置是否强制更新、暂停或回滚更新。 |
| 一级管理员 / 超级管理员 | 具备系统管理员全部能力，并可发布强制更新、扩大灰度范围、查看全量更新事件。 |

更新包发布属于系统级能力，不按部门授权范围分发 Skill 的规则处理。可选支持按渠道或灰度比例控制曝光，但不按普通 Skill 权限审批链路发布。

## 3. 版本与发布对象

客户端版本遵循 SemVer，格式为 `MAJOR.MINOR.PATCH`，例如 `1.6.0`。服务端保存的每个发布对象至少包含：

| 字段 | 说明 |
| --- | --- |
| `version` | 客户端版本号，必须递增。 |
| `buildNumber` | 构建号或提交号，用于排查构建来源。 |
| `platform` | 平台，首期固定为 `windows`。 |
| `arch` | 架构，首期固定为 `x64`。 |
| `channel` | 发布渠道，默认 `stable`，可扩展 `internal` / `beta`。 |
| `packageName` | 安装包文件名。 |
| `sizeBytes` | 安装包大小。 |
| `sha256` | 安装包 SHA-256。 |
| `signatureStatus` | 签名校验状态，至少区分 `signed` / `unsigned`。 |
| `releaseNotes` | 面向用户的更新说明。 |
| `mandatory` | 是否强制更新。 |
| `minSupportedVersion` | 最低可继续使用远端服务的客户端版本。 |
| `rolloutPercent` | 灰度比例，默认 100。 |
| `publishedAt` | 发布时间。 |
| `status` | `draft` / `published` / `paused` / `yanked`。 |

版本规则：

- 同一 `platform + arch + channel` 下，同一版本只能存在一条有效发布记录。
- `published` 版本可被客户端检测到；`draft`、`paused`、`yanked` 不作为新版本提示。
- 当 `currentVersion < minSupportedVersion` 时，服务端返回强制更新状态；客户端必须阻止新的远端写入操作，并提示用户先升级。
- 回滚不通过降低客户端版本实现，而是暂停或撤回问题版本，并将最新可用版本指向上一稳定版本。

## 4. 检测触发规则

客户端在以下时机检查更新：

- 登录成功并完成 `/desktop/bootstrap` 后立即检查一次。
- 已有会话恢复成功后检查一次。
- 用户打开设置模态时，如本次会话未检查或缓存已过期，则检查一次。
- 用户在设置中点击“检查更新”时立即检查。
- 网络从离线恢复为在线后检查一次。
- 长时间运行时按后台间隔检查，建议不少于 6 小时一次。

检查请求应携带：

- 当前客户端版本、构建号、平台、架构、操作系统版本。
- 当前登录用户、部门、设备 ID。
- 发布渠道，默认 `stable`。
- 上次提示或忽略的版本，用于服务端和客户端共同避免重复打扰。

检查结果分为：

| 结果 | 行为 |
| --- | --- |
| 无更新 | 设置中显示“已是最新版本”，不生成通知。 |
| 可选更新 | 生成通知，设置中展示更新卡片，用户可稍后处理。 |
| 强制更新 | 生成高优先级通知，设置中置顶展示，并限制发布、安装、审核、管理等远端写入动作。 |
| 当前版本不受支持 | 登录后允许进入最小界面，但阻止依赖远端服务的主要操作，要求先升级。 |
| 检查失败 | 不阻断登录；设置中展示最近一次检查失败原因和重试入口。 |

## 5. 用户提示与交互

### 5.1 顶栏通知

检测到新版本后，系统生成软件更新通知：

- 标题示例：`发现客户端新版本 1.6.0`
- 摘要示例：`包含稳定性修复和发布中心体验优化，建议尽快更新。`
- 点击行为：打开轻量更新提示弹窗。
- 已读行为：仅表示用户已读通知，不代表忽略该版本。

同一用户、同一设备、同一版本在未变化前不重复生成多条通知。若管理员将可选更新升级为强制更新，可以生成新的高优先级通知。

### 5.2 设置模态

设置模态必须展示当前客户端版本，并在存在更新时展示更新卡片：

- 当前版本。
- 最新版本。
- 更新类型：可选更新 / 强制更新。
- 更新说明。
- 包大小。
- 发布时间。
- 操作按钮：`立即更新`、`稍后提醒`、`查看详情`、`重新检查`。

如果用户已将可选版本选择“稍后提醒”，通知面板可以保持已读，但设置模态仍需保留更新提示，直到用户完成更新或管理员撤回该版本。

### 5.3 更新弹窗

更新弹窗显示：

- 新版本号和当前版本号。
- 更新说明。
- 下载进度、校验进度和失败重试。
- 安装前提醒：安装过程中可能需要关闭当前客户端。
- 强制更新时说明受限操作范围。

客户端不得在未获得用户确认时直接运行安装程序。

## 6. 用户更新流程

1. 用户登录成功，客户端检查更新。
2. 服务端返回可用版本。
3. 客户端写入本地更新缓存，并生成通知与设置卡片提示。
4. 用户点击“立即更新”。
5. 客户端请求短期有效的下载地址。
6. 客户端下载更新包到本地临时目录。
7. 下载完成后校验 SHA-256；如支持签名校验，则同时校验签名。
8. 校验通过后提示用户开始安装。
9. 用户确认后启动安装程序，并按操作系统流程完成安装。
10. 新版本启动后再次上报当前版本，更新提示消失。

失败处理：

- 下载失败：保留更新卡片，允许重试。
- Hash 或签名校验失败：删除本地临时包，记录安全错误，不允许安装。
- 安装被用户取消：保留更新卡片，不重复下载已校验通过的同一包，除非包已过期或被撤回。
- 更新包被管理员撤回：客户端下载前或安装前重新检查时不再允许继续安装该包。

## 7. 系统管理员推送流程

管理员推送客户端更新包的标准流程：

1. 在可信构建机或 CI 中构建 Windows x64 安装包。
2. 完成代码签名，生成 SHA-256 校验值和更新说明。
3. 将安装包复制到 Linux 服务器指定发布目录，或通过管理端上传。
4. 在 Docker 环境中执行发布命令或调用管理 API，将安装包写入对象存储，并创建 `draft` 发布记录。
5. 管理员检查版本号、Hash、签名状态、包大小、更新说明和目标渠道。
6. 先发布到 `internal` 或小比例灰度，验证检查、下载、安装和上报链路。
7. 灰度通过后发布到 `stable`，逐步扩大 `rolloutPercent` 至 100。
8. 发现问题时将版本置为 `paused` 或 `yanked`，客户端停止提示该版本。

管理员不得在生产 Linux 服务器上临时打包客户端，不得直接把文件放入 Nginx 静态目录绕过服务端元数据、Hash 和权限校验。

## 8. 离线与异常场景

- 离线登录或会话恢复时，不检查更新，不阻断本地已安装 Skill 使用。
- 恢复网络后立即检查更新。
- 更新检查失败不影响本地能力，但强制更新策略在下一次成功检查后生效。
- 下载地址必须短期有效，过期后客户端重新申请。
- 客户端本地缓存只保存元数据和临时安装包路径，不把更新包视为长期可信源。
- 用户设备时间不可信，强制更新判断以服务端时间和服务端返回为准。

## 9. 验收标准

- 旧版本客户端登录成功后能自动检查并提示新版本。
- 顶栏通知和设置模态均能展示同一个可用版本，并避免重复通知刷屏。
- 用户可手动检查更新，并看到无更新、检查失败、发现更新三类状态。
- 下载完成后必须校验 SHA-256；校验失败时不能启动安装。
- 强制更新时，客户端阻止远端写入型操作，并给出明确升级提示。
- 管理员能在 Linux Docker 环境中发布、暂停、撤回更新包。
- 灰度比例、渠道和版本撤回能影响客户端检查结果。
- 新版本安装并启动后，客户端不再提示已安装版本。
